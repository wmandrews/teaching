Workflow and File Organization

1. Create a project folder. Name it something that makes sense, and is web friendly:

For a midterm project, I'd call the folder

'midterm'

And I'd put it in my

'jour354' or 'jour654'

folder, which you should create at the beginning of a project (or class, as in this case.)

2. Grab an HTML template and drop it in the folder. Rename it as

'index.html'

Think of your browser like a file browser. You use slashes '/' to go into folders. Browsers read any file named 'index' as the file to show by default when looking into that folder.

This means we can have nice and neat urls like:

http://www.wilsonandrews.org/jour354/

Without having that messy filename.html on the end. Makes your urls a lot cleaner. You won't always have to use index.html files, but I always opt for that approach.

3. If I know I'm going to have javascript, css and images, which is almost always the case, I create folders for these.

js css images

You can call these folders whatever you like. Short is nice because you often have to type these things many times. A lot of people use imgs instead for that reason.

We've already seen how to refer to a folder from an html file.

<link rel="stylesheet" href="css/style.css" />
<script type="text/javascript" src="js/scripts.js"></script>
<img src="images/myimage.jpg" />

A COUPLE TRICKY THINGS

Sometimes you want to refer to the images folder from your css or js files. Let's use the example of a background image in css.

CSS

body {
    background:url(../images/hash.gif);
}

Because the css file is within a folder that is on the same level as the html file that we're referencing it from, the file reference must go up a level.

body {
    background:url(images/hash.gif);
}
/*
This would be looking for the image like this: css/images/hash.gif, since the css file is INSIDE the css folder
*/

We use ../ to go up a level, which puts us in the parent folder, midterm, before looking for images/hash.gif.

../images/hash.gif

However, in JavaScript, your code is actually being read by the browser from within the html page, so you can treat any image or other file references in your js as if they were in the html file.

JavaScript

var myImage = 'images/awesome.jpg';
$('#wrapper').html('<img src="' + myImage +'">');

See how this reference images/awesome.jpg is the same as if we were targeting the image from the html file?

If you had a data file that you were loading into your js, the same rule applies. Create a folder named data or something similar, and drop your data file in there. Then...

var myData = 'data/sample.xml';

We'll get into loading data in an upcoming class.

Publishing

Once you're done with creating all your files, and making your project work, it's time to publish.

The web is set up exactly the same as the filesystems on your computer. There are main folders, called root folders that host the index file at the root of the site, the top-level parent.

The root of the Washington Post's site is: http://www.washingtonpost.com/

The hosting server is set up with a root folder that has an index file in it. That index file is what you see when you visit the homepage.

Now on a huge site like that, the file system is a little more complex because it's powered by a CMS that dynamically creates pages instead of a human manually editing then publishing an HTML file like you guys have been doing. But it still uses the same structure as any other web site: files inside folders inside folders.

To publish to your own personal server, you can use an FTP client, and it's just like copying and pasting your files into a new folder. The only difference is that you're moving your files into a public folder that is accessible via a browser on the web.
